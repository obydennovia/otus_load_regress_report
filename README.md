# Автоматизация регрессионного нагрузочного тестирования WebTours (OTUS)

Репозиторий содержит готовую автоматизированную экосистему на базе **k6**, **InfluxDB** и **Grafana** для проведения ступенчатого нагрузочного тестирования (Step Load Test), выявления регрессии производительности и автоматического расчета максимальной полезной емкости системы (Maximum Useful Capacity).

---

## 📌 Цель проекта
Проведение сравнительного анализа производительности двух версий веб-приложения WebTours:
1. **Эталонный контур (Baseline):** `http://webtours.load-test.ru:1080`
2. **Регрессионный контур (Regression):** `http://webtours.load-test.ru:1090`

Цель — определить точку перегиба (деградации) нового релиза в условиях ступенчатого роста интенсивности, проверить соблюдение жестких критериев качества (SLA) и сформировать релизное заключение.

---

## 🏗️ Архитектура и стек технологий
* **Генератор нагрузки:** `k6 (by Grafana)` — запуск параллельных сценариев в рамках одного файла, прецизионное управление интенсивностью через `ramping-arrival-rate`.
* **Хранилище метрик:** `InfluxDB v1.x (локально)` — сбор посекундных агрегатов времени отклика и ошибок по протоколу Influx Line Protocol.
* **Визуализация:** `Grafana` — построение совмещенных аналитических графиков Time Series для моментального выявления расхождения версий.
* **Аналитический модуль:** `Node.js (JavaScript)` — автоматический парсинг базы данных, расчет перцентилей последнего теста и генерация финального отчета по регламенту.

---

## 📈 Профиль нагрузки и требования ТЗ

### 1. Run Logic и пропорция бизнес-операций:
Действия пользователей следуют в строгом логическом порядке. В рамках одной итерации k6 (1 запуск скрипта) инкапсулирована жесткая пропорция операций **1:2** согласно требованиям к АС:
* `01_Open_Login_Page` + `02_Submit_Login` (1 вход в систему)
* `03_Navigate_To_Flight_Form` + `04_Search_Flights_Submit` + `05_Select_Flight_From_List` + `06_Submit_Payment_Confirmation` (**2 цикла покупки билетов подряд**)

### 2. Сценарий ступенчатого разгона (70 минут, `timeUnit: '1m'`):
* **100% Базового профиля:** 60 итераций в минуту (в пересчете на запросы: $60 \times 3 = 180$ операций в минуту, или **10 800 ОП/Ч**).
* **Шаг разгона:** На каждой следующей ступени добавляется **+10% от базового профиля** (+6 итераций в минуту). Каждая ступень удерживается ровно **10 минут**.
* **Завершение:** На 70-й минуте (при 170% нагрузки / 18 360 ОП/Ч) происходит мгновенная остановка всех виртуальных пользователей сразу.

### 3. Критерии качества (SLA):
* **SLA по времени отклика (RT):** 95-й перцентиль (p95) каждого пользовательского действия не должен превышать **3 секунды (3000 мс)**.
* **SLA по надежности (Error Rate):** Не более **1.0%** неуспешных HTTP-ответов на ступени.

## 📂 Структура файлов проекта
* `webtours_step_max.js` — Скрипт k6. Содержит сценарии `baseline_max_test` и `regression_max_test`. Автоматически очищает куки и распределяет нагрузку на порты 1080 и 1090.
* `compare_k6_metrics.js` — Аналитический скрипт Node.js. Находит в InfluxDB точное время окончания последнего теста, изолирует его, рассчитывает утилизацию и формирует текстовый отчет.
* `webtours_users.json` — Пул тестовых учетных записей для параметризации авторизации.

## 🚀 Инструкция по запуску и анализу

### Шаг 1. Запуск базы данных
Убедитесь, что сервер InfluxDB запущен локально на порту `8086`. Создайте целевую базу данных через CLI Influx:
```powershell
CREATE DATABASE otus_webtours
```

### Шаг 2. Запуск нагрузочного тестирования
Запустите объединенный k6-скрипт. Метрики параллельно и в реальном времени начнут транслироваться в общую базу данных:
```powershell
k6 run --out influxdb=http://localhost:8086/otus_webtours webtours_step_max.js
```

### Шаг 3. Анализ в Grafana
Для визуального анализа создайте панель типа **Time Series** со следующим InfluxQL-запросом (с обязательной группировкой по `scenario`):
```sql
SELECT percentile("value", 95) FROM "http_req_duration" WHERE "expected_response" = 'true' AND \(timeFilter GROUP BY time(\)__interval), "scenario"
```
*В настройках панели справа (Graph styles) переключите `Connect null values` в положение **Always**, чтобы линии графиков не рвались.*

### Шаг 4. Автоматическая генерация релизного заключения
Сразу после завершения или прямо во время прогона теста выполните в терминале VS Code:
```bash
node compare_k6_metrics.js
```

Скрипт выведет в консоль готовое **Заключение релизного нагрузочного тестирования** с расчетом деградации, анализом рисков, списком критических дефектов и краткими выводами в формате:
* **Эталонный тест (:1080):** Максимальная производительность в рамках SLA составляет 18360 операций в час.
* **Регрессионный тест (:1090):** Максимальная производительность в рамках SLA составляет 11880 операций в час.
* **Сравнение версий:** Выявлено падение пропускной способности стабильной работы релиза на 35%. Точка перегиба пройдена на шаге №3 (120% профиля), где время отклика регрессионного стенда составляет 4129 мс относительно 224 мс эталона.
* **Деградация производительности:** Начиная со ступени в 120% профиля зафиксирована тотальная деградация производительности нового релиза.
