# Автоматизация регрессионного нагрузочного тестирования WebTours (OTUS)

Репозиторий содержит готовую автоматизированную экосистему на базе **k6**, **InfluxDB** и **Grafana** для проведения ступенчатого нагрузочного тестирования (Step Load Test), выявления регрессии производительности и автоматического расчета максимальной полезной емкости системы (Maximum Useful Capacity).

---

## 📌 Цель проекта
Проведение сравнительного анализа производительности двух версий веб-приложения WebTours:
1. **Эталонный контур (Baseline):** `http://webtours.load-test.ru:1080`
2. **Регрессионный контур (Regression):** `http://webtours.load-test.ru:1090`

**Описание:** Автоматизация процесса регрессионного нагрузочного тестирования веб-приложения WebTours. Проект направлен на выявление деградации производительности между релизами (сравнение текущих метрик времени отклика с эталонными значениями) и автоматическое формирование аналитического заключения на основе собранных данных.

**Цель** — определить точку перегиба (деградации) нового релиза в условиях ступенчатого роста интенсивности, проверить соблюдение жестких критериев качества (SLA) и сформировать релизное заключение.

---

## 🏗️ Архитектура и стек технологий
Система построена на базе контейнеризированной инфраструктуры и состоит из следующих компонентов:
* **k6** — инструмент для генерации нагрузки и выполнения тестовых сценариев.
* **InfluxDB (v1.8)** — временная база данных (Time Series Database) для оперативного сбора и хранения сырых метрик k6.
* **Grafana** — платформа для визуализации метрик и мониторинга хода тестирования в реальном времени.
* **Node.js** — аналитический модуль для постобработки результатов, расчета деградации и генерации финального отчета.
* **Docker / Docker Compose** — оркестрация и развертывание компонентов инфраструктуры.

## 📈 Профиль нагрузки и требования ТЗ

### 1. Run Logic и пропорция бизнес-операций:
Действия пользователей следуют в строгом логическом порядке. В рамках одной итерации k6 (1 запуск скрипта) инкапсулирована жесткая пропорция операций **1:2** согласно требованиям к АС:
* `01_Open_Login_Page` + `02_Submit_Login` (1 вход в систему)
* `03_Navigate_To_Flight_Form` + `04_Search_Flights_Submit` + `05_Select_Flight_From_List` + `06_Submit_Payment_Confirmation` (**2 цикла покупки билетов подряд**)

### 2. Сценарий ступенчатого разгона (70 минут, `timeUnit: '1m'`):
* **100% Базового профиля:** 60 итераций в минуту (в пересчете на запросы: $60 \times 3 = 180$ операций в минуту, или **10 800 ОП/Ч**).
* **Шаг разгона:** На каждой следующей ступени добавляется **+10% от базового профиля** (+6 итераций в минуту). Каждая ступень удерживается ровно **10 минут**.
* **Завершение:** На 70-й минуте (при 170% нагрузки / 18 360 ОП/Ч) происходит мгновенная остановка всех виртуальных пользователей сразу.

### 3. Критерии качества (SLA):
* **SLA по времени отклика (RT):** 95-й перцентиль (p95) каждого пользовательского действия не должен превышать **3 секунды (3000 мс)**.
* **SLA по надежности (Error Rate):** Не более **1.0%** неуспешных HTTP-ответов на ступени.

## 🚀 Быстрый старт

### 1. Запуск инфраструктуры (InfluxDB + Grafana)
Разверните контейнеры базы данных и системы визуализации из корневой директории проекта:
```bash
docker compose up -d
```
*База данных `otus_webtours` инициализируется в контейнере автоматически.*

### 2. Запуск нагрузочного теста (k6)
Запустите скрипт тестирования для отправки метрик в поднятый контейнер InfluxDB.

**Вариант А (Если k6 установлен локально):**
```bash
k6 run --out influxdb=http://localhost:8086/otus_webtours webtours_step_max.js
```

**Вариант Б (Запуск k6 через Docker):**
```bash
docker run --rm --network=host -i grafana/k6 run --out influxdb=http://localhost:8086/otus_webtours - <webtours_step_max.js
```

### 📊 3. Настройка мониторинга в Grafana
1. Откройте интерфейс: `http://localhost:3000` (Вход по умолчанию: `admin` / `admin`).
2. Перейдите в **Connections** -> **Data Sources** -> **Add data source** -> **InfluxDB**.
3. Укажите параметры подключения:
   - **URL**: `http://influxdb:8086`
   - **Database**: `otus_webtours`
   - **HTTP Method**: `GET`
4. Нажмите **Save & test**.
5. Создайте панель **Time Series** со следующим InfluxQL-запросом для вывода перцентилей:
   ```sql
   SELECT percentile("value", 95) FROM "http_req_duration" WHERE "expected_response" = 'true' AND \(timeFilter GROUP BY time(\)__interval), "scenario", "group"
   ```
6. В поле **Alias by** снизу введите: `$tag_scenario | $tag_group`
7. В правых настройках панели (**Graph styles**) найдите пункт **Connect null values** и переведите его в положение **Always**.

### 📈 4. Анализ полученного графика нагрузки в Grafana

После завершения теста на созданном графике отобразится динамика изменения 95-го перцентиля времени отклика (`http_req_duration`) по каждому сценарию. При анализе графика обращайте внимание на следующие ключевые маркеры:
1. **Поиск точки излома (Ступени нагрузки):**
   - Сравнивайте рост количества виртуальных пользователей (VUs) с временем отклика.
   - **Норма**: Время отклика остается стабильным (график горизонтальный) или растет незначительно при переходе на новые ступени нагрузки.
   - **Деградация**: Резкий излом графика вверх на определенной ступени указывает на то, что система достигла предела производительности.

2. **Идентификация «тяжелых» сценариев:**
   - Благодаря группировке `GROUP BY "scenario"`, каждая цепочка шагов (например, `оплата`, `поиск билетов`, `авторизация`) получит свой цвет на графике.
   - Определите, какой именно сценарий демонстрирует наибольший рост времени отклика под нагрузкой — это ваша главная точка оптимизации.

3. **Сгущение точек и разрывы (Connect null values):**
   - Опция `Connect null values: Always` сглаживает провалы в периоды, когда запросы по конкретному сценарию не отправлялись, позволяя видеть непрерывный тренд производительности.

4. **Интерпретация результатов для ТЗ:**
   - Если линия 95-го перцентиля на максимальной ступени нагрузки уходит выше допустимого порога SLA (например, > 3000 мс) — тест считается неуспешным. Требуется локализовать узкое место (Database, CPU, Network).

5. Для сравнительного анализа регрессии выполните изоляцию метрик:
   - Найдите в легенде транзакцию эталонного теста `baseline_max_test | ::03_Flight_Transaction` и **кликните по ней** — на графике останется только она.
   - Зажмите клавишу **Ctrl** (или **Cmd** на Mac) и **кликните по транзакции нового релиза** `regression_max_test | ::03_Flight_Transaction`.
   - На графике останутся только две эти линии для наглядного сравнения деградации (поиск аномалий и оценка роста времени отклика под нагрузкой).

### 🤖 5. Генерация аналитического отчета регрессии
Скрипт `compare_k6_metrics.js` является самостоятельным и использует встроенные сетевые модули Node.js.

### 🤖 5. Генерация аналитического отчета регрессии
Для работы скрипта анализа необходима библиотека `axios`. Перед первым запуском инициализируйте окружение и установите зависимости:

```bash
# Инициализация проекта и установка axios
npm init -y
npm install axios

# Запуск генерации отчета
node compare_k6_metrics.js
```

Скрипт считает метрики из локальной базы InfluxDB и выведет в консоль готовое **Заключение релизного нагрузочного тестирования** с расчетом деградации, анализом рисков, списком критических дефектов и краткими выводами в формате:
* **Эталонный тест (:1080):** Максимальная производительность в рамках SLA составляет 18360 операций в час.
* **Регрессионный тест (:1090):** Максимальная производительность в рамках SLA составляет 11880 операций в час.
* **Сравнение версий:** Выявлено падение пропускной способности стабильной работы релиза на 35%. Точка перегиба пройдена на шаге №3 (120% профиля), где время отклика регрессионного стенда составляет 4129 мс относительно 224 мс эталона.
* **Деградация производительности:** Начиная со ступени в 120% профиля зафиксирована тотальная деградация производительности нового релиза.
